2.1.- Clase Abstracta vs Interfaz en Java

Clase Abstracta

Una clase abstracta es una clase que no puede ser instanciada directamente, sino que debe ser extendida por otras clases. Puede contener:

  • Métodos abstractos (sin implementación)
  • Métodos concretos (con implementación)
  • Campos (variables)
  • Constructores
  • Métodos estáticos y finales

Se usa cuando las clases hijas comparten una relación "es-un" y tienen funcionalidad común.

Interfaz

Una interfaz es una colección de métodos abstractos (a partir de Java 8 también puede tener métodos default y estáticos). Las interfaces:

  • Solo pueden contener constantes (public static final)
  • Métodos abstractos (antes de Java 8)
  • No pueden tener constructores
  • No pueden tener campos de instancia
  • A partir de Java 8 pueden tener métodos default y estáticos con implementación

Se usan para definir contratos que las clases deben implementar, permitiendo múltiples herencias de comportamiento.

Ejemplo en Java
// Interfaz
interface PuedeNadar {
    void nadar(); // método abstracto
    
    default void respirarBajoAgua() { // método default (Java 8+)
        System.out.println("Tomando aire bajo el agua");
    }
}

// Clase abstracta
abstract class Animal {
    private String nombre;
    
    public Animal(String nombre) {
        this.nombre = nombre;
    }
    
    public String getNombre() {
        return nombre;
    }
    
    abstract void hacerSonido(); // método abstracto
    
    public void dormir() { // método concreto
        System.out.println(nombre + " está durmiendo");
    }
}

// Clase concreta que extiende Animal e implementa PuedeNadar
class Ballena extends Animal implements PuedeNadar {
    public Ballena(String nombre) {
        super(nombre);
    }
    
    @Override
    void hacerSonido() {
        System.out.println(getNombre() + " hace sonidos de ballena");
    }
    
    @Override
    public void nadar() {
        System.out.println(getNombre() + " está nadando majestuosamente");
    }
}

public class Main {
    public static void main(String[] args) {
        Ballena moby = new Ballena("Moby Dick");
        moby.hacerSonido(); // método abstracto implementado
        moby.dormir();      // método heredado de Animal
        moby.nadar();       // método de la interfaz
        moby.respirarBajoAgua(); // método default de la interfaz
    }
}
Salida esperada:
Moby Dick hace sonidos de ballena
Moby Dick está durmiendo
Moby Dick está nadando majestuosamente
Tomando aire bajo el agua
Cuando usar cada una:

Usa clase abstracta cuando:

  • Quieres compartir código entre varias clases relacionadas
  • Necesitas declarar campos no estáticos o no finales
  • Quieres definir constructores

Usa interfaz cuando:

  • Quieres definir un contrato para clases no relacionadas
  • Necesitas múltiples herencias de tipo
  • Quieres especificar comportamiento sin implementación (antes de Java 8)
  • Quieres añadir funcionalidad a clases existentes sin modificar su herencia

2.2.- Ejemplo de Clase Abstracta: Sistema de Cuentas Bancarias

Implementación de un sistema de cuentas bancarias usando una clase abstracta Cuenta y una clase concreta CuentaBasica.

Cuenta.java (Clase abstracta)
public abstract class Cuenta {
    private String nombre;
    private String no_cuenta;
    private double saldo;

    public void defineNombre(String nom) {
        nombre = nom;
    }

    public void defineNoCuenta(String no_cta) {
        no_cuenta = no_cta;
    }

    public void defineSaldoInicial(double sal) {
        saldo = sal;
    }

    public void incrementaSaldo(double valor) {
        saldo = saldo + valor;
    }

    public double informaSaldo() {
        return saldo;
    }

    public void imprimeEstadoCta() {
        System.out.println("Titular de la cuenta: " + nombre);
        System.out.println("Número de la cuenta: " + no_cuenta);
        System.out.println("Saldo de la cuenta: " + saldo);
    }

    public abstract void generaInteres(int mes);
}
CuentaBasica.java (Clase concreta)
public class CuentaBasica extends Cuenta {
    public void generaInteres(int mes) {
        double interes = super.informaSaldo();
        interes = interes * 0.01;
        super.incrementaSaldo(interes);
    }
}
Main.java (Ejecución)
public class Main {
    public static void main(String[] args) {
        CuentaBasica cuenta1 = new CuentaBasica();
        cuenta1.defineNombre("Alberto Martinez Pacheco");
        cuenta1.defineNoCuenta("001");
        cuenta1.defineSaldoInicial(1000);
        
        System.out.println("Corrida de interes Anual:");
        for(int x = 1; x <= 12; x++) {
            System.out.println("MES " + x);
            System.out.println("*************************" + x);
            cuenta1.generaInteres(x);
            cuenta1.imprimeEstadoCta();
        }
    }
}

3.2.- Diagrama de Secuencia: Avance

Definición del Problema

En el local comercial objeto de estudio, existen cuatro cajas registradoras destinadas al proceso de cobro a clientes. Las cajas 1, 2 y 3 están interconectadas a un mismo servidor central que gestiona las operaciones de venta, tanto en efectivo como con tarjeta, y todas deben realizar su corte diario de operaciones antes de las 12:00 de la noche. De forma similar, la caja 4, aunque independiente de las anteriores, también está sujeta al mismo requerimiento de cierre diario de efectivo y pagos con tarjeta antes de la medianoche.

El problema surge debido a que el proceso de cierre de caja implica temporalmente que dicha caja no esté disponible para realizar cobros, lo que reduce la cantidad de estaciones operativas en ese momento. En particular, si el cierre de múltiples cajas coincide o se realiza sin coordinación adecuada, se corre el riesgo de que ninguna caja esté disponible para atender a los clientes, provocando acumulación de personas en fila, retrasos en el servicio y potencial pérdida de ventas.

3.4.- Resumen de Diagramas UML

Resumen Ampliado de Diagramas UML

El Lenguaje Unificado de Modelado, conocido como UML (Unified Modeling Language), es una herramienta esencial en el ámbito de la ingeniería de software para visualizar, especificar, construir y documentar los artefactos de un sistema. Entre los diversos tipos de diagramas que ofrece UML, destacan seis por su relevancia en el análisis y diseño orientado a objetos: diagrama de objetos, diagrama de componentes, diagrama de actividades, diagrama de estados, diagrama de secuencias y diagrama de clases. Cada uno cumple un propósito específico y complementario dentro del proceso de desarrollo, permitiendo representar de manera gráfica distintos aspectos del comportamiento y la estructura del sistema.

1. Diagrama de Objetos

El diagrama de objetos es una representación estática que muestra instancias específicas de clases en un momento determinado. En otras palabras, mientras que el diagrama de clases representa el diseño general y abstracto de las entidades del sistema, el diagrama de objetos presenta un ejemplo concreto de cómo lucen dichas entidades durante la ejecución.

2. Diagrama de Componentes

El diagrama de componentes representa la estructura física del sistema, es decir, la organización y las dependencias entre los componentes de software. Cada componente se puede considerar una unidad lógica de implementación, como archivos ejecutables, bibliotecas, paquetes o módulos.

3. Diagrama de Actividades

El diagrama de actividades representa flujos de trabajo o procesos dentro del sistema. Se utiliza para modelar el comportamiento dinámico de un sistema, particularmente en lo que respecta a la lógica de control, decisiones, concurrencia y sincronización.

4. Diagrama de Estados

El diagrama de estados, también llamado diagrama de máquinas de estado, se utiliza para modelar los distintos estados por los que puede pasar un objeto a lo largo de su ciclo de vida, así como las transiciones que se producen entre esos estados debido a eventos específicos.

5. Diagrama de Secuencias

El diagrama de secuencias pertenece al conjunto de diagramas de interacción de UML y se enfoca en mostrar cómo interactúan los objetos entre sí a lo largo del tiempo. Representa el flujo de mensajes entre objetos para llevar a cabo una función o proceso en particular.

6. Diagrama de Clases

El diagrama de clases es uno de los más fundamentales en UML, ya que representa la estructura estática del sistema. Muestra las clases del sistema, sus atributos, métodos (operaciones) y las relaciones entre ellas, como asociaciones, herencia, agregación y composición.

Conclusión

Los diagramas UML constituyen una herramienta poderosa para el modelado de sistemas orientados a objetos. Cada tipo de diagrama cumple un papel específico que contribuye a una mejor comprensión, diseño y documentación del software. En conjunto, permiten al equipo de desarrollo visualizar tanto la estructura estática como el comportamiento dinámico del sistema.
